# 个人博客补全路线图

本文档把“个人知识博客 + 轻量互动”的产品定位，与后台管理补全方案合并成一套可实施路线图。目标是让博客从“能发布内容”升级为“可长期写作、可控互动、移动端可用、后台好维护、安全有闭环”的个人知识产品。

## 1. 产品定位

### 1.1 核心定位

本博客不是大型社区，也不是营销型站点。它的核心价值是：

- 沉淀高质量文章。
- 提供舒服稳定的阅读体验。
- 支持轻量、可审核、低负担的互动。
- 让作者能在桌面端和手机端稳定管理内容。
- 保证权限、安全、备份、部署流程有明确边界。

### 1.2 设计原则

- 文章优先：所有互动、后台、统计都服务于文章质量和维护效率。
- 后台高效：高频动作必须少步骤完成，例如发文、保存、审评论、改设置。
- 评论可控：默认审核，风险可见，批量处理，避免评论区成为维护负担。
- 移动端真实可用：手机端不是桌面缩小版，后台列表要卡片化，编辑弹窗要 sheet 化。
- 安全边界清晰：F12 干扰只是干扰项，不作为真实安全能力；真实安全依赖服务端权限、限流、HTTPS、密钥隔离和审计。
- 不泄露秘密：私钥、环境变量、部署密钥、备份包不进入 git，也不在后台明文展示。

## 2. 总体架构

### 2.1 当前技术边界

- 前台与后台 UI：Astro + React islands。
- 数据和认证：PocketBase。
- 静态搜索：Pagefind。
- 部署入口：Caddy / Docker。
- 安全增强：Caddy 规则、PocketBase rules、PocketBase hooks、后台权限守卫。

### 2.2 推荐分层

```text
浏览器
  ├─ 前台页面：阅读、搜索、评论、标签、归档
  ├─ 后台页面：文章、评论、标签、媒体、设置、安全、日志
  └─ React islands：登录、搜索、评论、后台管理交互

Astro 应用层
  ├─ layouts：BaseLayout / AdminLayout / PostLayout
  ├─ components：前台组件、后台组件、通用 UI
  ├─ hooks：认证、站点设置、评论、PocketBase
  ├─ lib：安全清理、PocketBase client、设置合并、格式化工具
  └─ scripts：构建检查、移动端检查、部署前检查

PocketBase 数据层
  ├─ posts / tags / post_tags
  ├─ comments / public_comments
  ├─ users / settings
  ├─ media_assets
  ├─ audit_logs
  ├─ reactions / comment_reports
  ├─ announcements / friend_links
  └─ post_versions

部署与安全层
  ├─ Caddy：HTTPS、安全头、扫描路径拦截、管理端白名单
  ├─ PocketBase rules：数据读写权限
  ├─ PocketBase hooks：角色保护、评论清理、邮件通知
  └─ scripts：迁移验证、备份、敏感文件检查、健康检查
```

### 2.3 后台架构原则

- 后台 Shell 独立于前台 Header，不复用前台导航逻辑。
- `AdminLayout` 负责整体布局、移动端 topbar、drawer 容器和页面安全区域。
- `AdminSidebar` 负责权限过滤、分组导航、桌面固定侧栏、移动端抽屉导航。
- 每个后台模块组件只负责本模块数据和交互，不在组件里散落全局权限规则。
- 权限判断前端只做体验优化，最终权限必须由 PocketBase rules / hooks 保证。
- 危险操作统一走确认组件，不直接使用原生 `confirm()` 作为最终方案。
- 所有后台写操作都应该产生可追踪结果：成功提示、失败提示、必要时写入操作日志。

## 3. 后台模块设计

### 3.1 后台首页仪表盘

目标：把 `/admin` 做成“今天该处理什么”的工作台。

应实现：

- 数据概览：文章总数、已发布、草稿、归档、标签数、评论数、待审核评论、用户数。
- 待办事项：待审评论、待完善草稿、安全警告、备份过期、部署检查失败。
- 最近活动：最近文章、最近评论、最近登录、最近设置变更、最近角色变更。
- 快捷入口：新建文章、审核评论、管理标签、站点设置、安全中心、查看前台。
- 角色提示：不同角色看到不同的“可做什么 / 不可做什么”。

移动端要求：

- 第一屏优先显示待办事项和快捷入口。
- 统计卡片后置，不压住关键操作。
- 最近活动使用卡片，不使用宽表格。
- 主要操作按钮触控高度不少于 44px。

### 3.2 文章管理

目标：让文章创作、维护、发布成为后台最顺的流程。

列表能力：

- 搜索标题、slug。
- 按状态筛选：草稿、已发布、归档、定时发布。
- 按标签、作者、发布时间、更新时间、浏览量筛选或排序。
- 只看置顶、只看精选。
- 桌面端表格，移动端卡片。

文章操作：

- 新建、编辑、保存草稿、发布、撤回、归档、复制、删除。
- 预览前台效果。
- 复制文章链接。
- 批量发布、批量撤回、批量归档、批量删除、批量添加标签。

编辑器能力：

- Markdown 编辑。
- 实时预览。
- 桌面端分屏，移动端编辑 / 预览 tabs。
- 自动保存草稿。
- 离开页面提醒。
- 本地草稿恢复。
- 图片插入、封面选择。
- SEO 设置折叠区。

文章质量检查：

- 标题、摘要、slug、正文不能为空。
- slug 格式合法且唯一。
- 建议添加封面、标签、SEO 描述。
- 检查正文过短、SEO 描述过短、无 alt 图片、超长链接。
- 发布前显示质量检查结果，不强制阻止低风险项。

移动端要求：

- 文章编辑弹窗改为全屏 sheet。
- 顶部 sticky：标题、关闭、状态。
- 中间滚动：正文表单。
- 底部 sticky：取消、保存、发布。
- 输入法弹出后保存按钮仍可访问。

### 3.3 评论管理

目标：让评论默认可控，并能快速处理。

队列：

- 待审核。
- 已通过。
- 垃圾评论。
- 被举报。
- 已删除或隐藏。

筛选：

- 按状态、文章、昵称、邮箱、IP、是否有链接、是否被举报筛选。

操作：

- 通过、拒绝、标记垃圾、删除、恢复、回复、查看所属文章。
- 批量通过、批量标记垃圾、批量删除。

风险提示：

- 同 IP 高频评论。
- 重复内容。
- 包含链接。
- 内容过短或过长。
- 邮箱格式异常。
- 命中敏感词。

移动端要求：

- 每条评论一张审核卡片。
- 正文可展开 / 收起。
- 操作按钮顺序固定：通过、垃圾、删除。
- 危险操作二次确认。
- 连续审核不跳页面，不打断队列。

### 3.4 标签管理

目标：让标签成为内容组织能力，而不是简单字符串。

字段：

- 名称。
- slug。
- 描述。
- 颜色。
- 排序。
- 是否推荐。

能力：

- 搜索标签。
- 按文章数量排序。
- 查看标签下文章。
- 合并标签。
- 删除前提示影响文章数量。
- 删除前可迁移到另一个标签。

移动端要求：

- 标签列表卡片化。
- slug 和描述允许换行。
- 颜色块固定尺寸，不参与文本宽度计算。

### 3.5 媒体库

目标：减少文章封面和正文图片依赖手填 URL。

字段：

- 文件。
- alt 文本。
- 上传者。
- 文件大小。
- 使用次数。
- 创建时间。

能力：

- 上传图片。
- 搜索图片。
- 复制 URL。
- 设为封面。
- 插入文章。
- 编辑 alt。
- 删除未使用图片。

安全要求：

- 限制文件类型。
- 限制文件大小。
- 不允许上传脚本文件。
- 图片 URL 不展示任何服务器敏感路径。

### 3.6 用户管理

目标：保持权限边界清楚，不做复杂社区用户系统。

角色：

- `reader`：普通读者，不能进入后台。
- `author`：管理自己的文章和标签。
- `admin`：管理所有文章、评论、标签、媒体。
- `super_admin`：管理用户、设置、安全、日志、备份。

能力：

- 搜索邮箱、昵称。
- 按角色、状态、注册时间筛选。
- 修改角色。
- 禁用 / 恢复用户。
- 查看用户评论。
- 查看用户操作日志。

保护规则：

- 最后一个 `super_admin` 不能降级或禁用。
- `admin` 不能提升其他人为 `super_admin`。
- 普通注册只能成为 `reader`。
- 禁用用户后 token 至少在下次刷新时失效。

### 3.7 设置中心

目标：把设置拆清楚，避免一个表单堆满所有配置。

建议分组：

- 站点设置：站点名、slogan、描述、logo、头像、作者、备案、社交链接。
- SEO 设置：默认 title、默认 description、默认 OG 图、robots、sitemap、RSS。
- 评论设置：开关、默认审核、回复开关、敏感词、邮件通知。
- 安全设置：F12 干扰、危险操作二次确认、登录策略说明、后台白名单说明。
- 主题设置：默认主题、是否允许切换、首页模块显示。
- 友链设置：可跳转到友链管理。
- 公告设置：可跳转到公告管理。

保存体验：

- 字段级校验。
- 保存中、保存成功、保存失败状态。
- 未保存离开提醒。
- 保存失败不只 `alert()`，要在当前表单附近展示错误。

### 3.8 安全中心

目标：从静态说明升级为“检查清单 + 风险面板”。

检查项：

- HTTPS。
- HSTS。
- CSP。
- 登录限流。
- 管理后台 IP 白名单。
- PocketBase 角色规则。
- 评论 XSS 清理。
- 用户角色写入保护。
- `.env` / 私钥 / 部署包忽略检查。
- `login_security.pb.js.disabled` 状态说明。

风险面板：

- 最近登录失败。
- 高频失败 IP。
- 高频失败邮箱。
- 最近管理员登录。
- 最近危险操作。

分级：

- 通过。
- 警告。
- 失败。
- 待配置。

安全文案边界：

- F12 干扰只能写成“减少普通误触或低水平调试”，不能写成防破解。
- 前端加密不能保护密钥，任何密钥都不能放前端。
- 真实安全边界必须是服务端权限、限流、HTTPS、密钥隔离、审计。

### 3.9 操作日志

目标：后期排查问题时能知道“谁在什么时候改了什么”。

记录字段：

- 操作者。
- 操作类型。
- 目标集合。
- 目标 ID。
- 操作摘要。
- IP。
- User-Agent。
- 时间。

记录动作：

- 创建文章、编辑文章、发布文章、删除文章。
- 审核评论、删除评论、恢复评论。
- 修改用户角色、禁用用户、恢复用户。
- 修改设置。
- 上传媒体、删除媒体。
- 登录成功、登录失败。

后台能力：

- 按用户筛选。
- 按动作筛选。
- 按时间筛选。
- 按对象搜索。
- 查看详情。

### 3.10 公告、友链、关于页

公告管理：

- 标题、内容、类型、启用状态、开始时间、结束时间。
- 类型：普通、提示、维护、重要。
- 前台顶部展示。
- 后台首页显示当前公告状态。

友链管理：

- 名称、URL、描述、头像、状态、排序。
- 前台友链展示。
- 可选：失效检测。

关于页管理：

- 标题、简介、作者介绍、技术栈、时间线、联系方式、社交链接。
- 后台编辑后前台展示。

### 3.11 备份与部署管理

目标：只显示运维状态，不泄露敏感信息。

能力：

- 上次备份时间。
- 备份文件大小。
- 备份是否成功。
- 保留数量。
- 手动备份入口，仅 `super_admin`。
- 最近部署时间。
- 健康检查状态。

边界：

- 后台不展示服务器私钥。
- 后台不展示完整环境变量。
- 备份下载需要二次确认，最好限制为本机或受信 IP。

## 5. 实现路线

### Phase 0：文档与现状整理

目标：先让团队和后续 agent 知道做什么、不做什么。

任务：

- 完成本文档。
- 对照现有文件列出已实现、半实现、未实现模块。
- 标记中文乱码文件和高风险临时文件。
- 确认当前未提交改动归属，不覆盖用户已有改动。

验收：

- 有完整路线图。
- 有明确优先级。
- 没有修改业务代码。

### Phase 1：后台地基修复

任务：

- 修复后台和文档中的中文乱码。
- 整理 README、部署文档、回滚文档。
- 固化本地检查命令：build、mobile check、migration check。
- 添加敏感文件检查说明或脚本。

验收：

- 后台文案正常显示。
- 文档能被中文阅读。
- 私钥、`.env`、部署包不进入 git。

### Phase 2：后台移动端闭环

任务：

- `AdminLayout` 增加移动端 topbar。
- `AdminSidebar` 拆分桌面固定侧栏与移动端 drawer。
- 后台文章、评论、标签、用户、设置、安全页移动端卡片化。
- 全局后台 modal / drawer 支持 safe area、焦点管理、滚动锁定。

验收：

- 360px 下 `/admin`、`/admin/posts`、`/admin/comments`、`/admin/settings` 无横向滚动。
- 手机端能新建 / 编辑 / 保存文章。
- 手机端能审核评论。

### Phase 3：文章创作增强

任务：

- 编辑器增加 Markdown 预览。
- 增加自动保存、草稿恢复、离开提醒。
- 增加文章质量检查。
- 增加 SEO 字段。
- 增加置顶、精选。

验收：

- 作者能在后台完整写作并预览。
- 发布前能看到质量检查。
- SEO 字段能保存并用于前台 meta。

### Phase 4：评论可控增强

任务：

- 评论审核队列重构。
- 增加批量通过、垃圾、删除。
- 增加风险提示。
- 增加管理员回复标识。
- 可选增加评论举报。

验收：

- 待审核评论能连续处理。
- 长评论、长邮箱、长 IP 不撑破布局。
- 删除和垃圾操作有确认。

### Phase 5：前台阅读增强

任务：

- 文章详情增加 TOC、阅读进度、上一篇 / 下一篇、相关文章。
- 首页增加精选、最近更新、热门文章。
- 标签页升级。
- 搜索高亮和筛选增强。
- 404、空状态、错误状态补齐。

验收：

- 文章页阅读体验完整。
- 首页能引导到最新和精选内容。
- 搜索失败和无结果可理解。

### Phase 6：后台产品化

任务：

- 仪表盘升级。
- 设置中心拆分。
- 操作日志。
- 媒体库基础版。
- 友链、公告、关于页管理。

验收：

- 后台常用维护动作都有入口。
- 关键写操作有日志。
- 站点展示内容不必频繁改代码。

### Phase 7：安全与运维闭环

任务：

- 安全中心真实检查项。
- 备份脚本和备份状态。
- 部署前检查脚本。
- 线上健康检查清单。
- 回滚流程文档。

验收：

- 上线前能执行固定检查。
- 上线后能验证首页、登录、文章、后台、API。
- 出问题能回滚前端构建，不误动数据卷。

## 6. 自我审查机制

每次开发完成后，必须按以下维度自查。

### 6.1 产品定位自查

- 是否仍然服务于“个人知识博客 + 轻量互动”？
- 是否把评论做得过重，变成社区产品？
- 是否为了装饰牺牲阅读和后台效率？
- 是否让作者的发文、审评论、改设置更简单？

### 6.2 权限自查

- 前端隐藏入口之外，PocketBase rules 是否也限制了权限？
- `reader` 是否无法进入后台？
- `author` 是否不能管理用户和设置？
- `admin` 是否不能提升 super_admin？
- 最后一个 super_admin 是否被保护？

### 6.3 安全自查

- 是否把任何密钥、私钥、token、服务器密码放进代码或文档？
- 是否把 F12 干扰误描述为真实安全？
- 是否新增了前端可见密钥？
- 登录、评论、后台写操作是否有服务端限制？
- 危险操作是否有二次确认和操作日志？

### 6.4 移动端自查

- 320 / 360 / 390 / 430 宽度是否无页面级横向滚动？
- 后台 drawer 打开后 body 是否锁定滚动？
- modal / sheet 关闭后焦点是否能回到触发按钮？
- 输入法弹出时保存按钮是否还能访问？
- 主要按钮是否至少 44px 高？

### 6.5 内容质量自查

- 文章标题、摘要、slug 是否必填？
- SEO 描述是否可维护？
- 图片是否有 alt？
- 代码块、表格、长链接是否不会撑破页面？
- 相关文章、上一篇 / 下一篇是否不会引用草稿或归档文章？

### 6.6 可维护性自查

- 是否复用了现有组件和 hooks？
- 是否把权限规则散落在多个组件里？
- 是否引入了不必要的大依赖？
- 是否保留了桌面端已有高效体验？
- 是否没有覆盖用户未提交改动？

## 10. 完成标准

当以下条件满足时，可以认为第一阶段补全完成：

- 手机端能登录后台。
- 手机端能发文章、保存文章、审核评论、修改设置。
- 桌面端文章和评论管理效率不下降。
- 评论默认可控，不会绕过审核直接公开。
- 后台危险操作都有确认。
- 关键后台写操作可审计。
- 文章详情具备基本 SEO 和阅读增强。
- 360px 下核心前台和后台页面无页面级横向滚动。
- `npm run build` 成功。
- PocketBase 迁移验证成功。
- 私钥、`.env`、部署包不进入 git。


## 11. 前台阅读与内容呈现补全

后台补全的最终目的是让前台内容更好读、更容易被发现。因此前台也需要形成完整的信息架构。

### 11.1 首页模块

建议首页模块按优先级排列：

- 顶部识别区：站点名、简短介绍、主要入口。
- 精选文章：由 `is_featured` 控制，展示 3 到 6 篇。
- 最近更新：按 `published_at` 或 `updated` 排序。
- 热门文章：按浏览量或近 7 天热度排序。
- 标签入口：展示推荐标签和文章数量。
- 最近评论：只展示已通过评论，隐藏邮箱和 IP。
- 作者卡片：头像、简介、社交链接。
- 友链入口：展示部分启用友链。

移动端策略：

- 首屏必须能看到下一段内容的开头，避免 hero 占满整屏。
- 首页模块之间使用稳定间距，不使用过多装饰卡片。
- 热门文章和最近评论在手机端后置，避免干扰阅读入口。

### 11.2 文章详情

建议补齐：

- 目录 TOC：从 `h2` / `h3` 生成。
- 阅读进度条：顶部细条或侧边进度。
- 预计阅读时间：优先由后台保存字段，否则前端计算。
- 上一篇 / 下一篇：只引用已发布文章。
- 相关文章：同标签优先，再按发布时间补足。
- 代码块复制：复制成功有轻提示。
- 图片预览：可选，不影响正文阅读。
- 文章反应：轻量按钮，例如“有用”“喜欢”“思考”。
- 分享按钮：复制链接优先，其他平台可后置。

边界：

- 不引入复杂富文本编辑器导致包体膨胀。
- 不为了互动让文章详情变成社交流。
- 代码块、表格、长链接只允许局部横向滚动，不允许撑宽整页。

### 11.3 归档与标签

归档页：

- 按年份分组。
- 年份下按月份排序。
- 显示每月文章数量。
- 支持快速跳转年份。

标签页：

- 标签云。
- 推荐标签。
- 标签描述。
- 标签文章数量。
- 标签详情页展示该标签下文章。

### 11.4 搜索体验

搜索能力：

- 关键词高亮。
- 按标签筛选。
- 搜索结果显示标题、摘要、标签、发布时间。
- 无结果时提供返回全部文章和热门标签入口。
- Pagefind 加载失败时显示明确错误，不空白。

移动端：

- 搜索面板全屏化。
- 输入框 sticky 在顶部。
- 结果区域独立滚动。
- 关闭后恢复焦点。

## 12. 内容运营能力

这些功能不复杂，但能显著降低长期维护成本。

### 12.1 内容日历

用途：帮助作者管理写作节奏。

能力：

- 草稿计划发布时间。
- 月视图展示草稿、已发布、定时发布。
- 临近计划发布时在仪表盘提醒。
- 只作为后台辅助，不强制改变文章发布流程。

### 12.2 写作任务清单

每篇草稿可以维护轻量 checklist：

- 标题完成。
- 摘要完成。
- 正文完成。
- 图片检查。
- SEO 检查。
- 标签检查。
- 预览检查。

这可以先存在文章字段中，也可以后续拆成独立集合。

### 12.3 系列与专题

适合知识博客的长期沉淀。

能力：

- 新增 `series` 集合。
- 文章可关联一个或多个系列。
- 系列页展示简介、文章顺序、学习路径。
- 后台支持拖拽或数字排序。

第一阶段可先不做，但数据结构设计时应预留。

### 12.4 外链与引用管理

文章中常会有参考资料。

可选能力：

- 后台记录文章引用链接。
- 检查外链是否失效。
- 前台文章末尾展示参考资料。
- 对外链自动加 `rel="noopener noreferrer"`。

## 13. 轻互动深化

互动必须轻，不做复杂关系链。

### 13.1 文章反应

推荐反应类型：

- 有用。
- 喜欢。
- 受启发。

规则：

- 未登录用户可通过匿名 fingerprint 参与，但要限频。
- 登录用户按用户 ID 记录。
- 同一用户 / fingerprint 对同一文章同一类型只能点一次。
- 后台能看到每篇文章反应数量。

### 13.2 评论举报

规则：

- 举报只进入后台，不自动删除。
- 多次举报可提高风险等级。
- 后台评论卡片显示举报数。
- 举报理由使用固定选项加补充说明。

### 13.3 邮件通知

通知类型：

- 新评论通知管理员。
- 评论通过通知评论者，可选。
- 管理员回复通知评论者。
- 站点异常或备份失败通知超级管理员。

实现边界：

- 邮件发送应在 PocketBase hooks 或服务端执行。
- 前端不能暴露 SMTP 密钥。
- 通知失败不应阻塞评论保存，但要记录错误。

### 13.4 互动限流

需要限制：

- 评论提交。
- 评论举报。
- 文章反应。
- 登录尝试。

限流位置：

- 前端只做体验提示。
- 服务端或反向代理必须有真实限制。

## 14. 数据统计与分析

统计应以“帮助写作决策”为目标，不追求复杂 BI。

### 14.1 文章统计

可统计：

- 总浏览量。
- 今日浏览量。
- 近 7 天浏览量。
- 评论数量。
- 反应数量。
- 搜索命中次数，可选。

后台展示：

- 热门文章。
- 最近增长快的文章。
- 长期无人阅读文章，可用于归档或重写。

### 14.2 评论统计

可统计：

- 待审核数量。
- 通过率。
- 垃圾评论数量。
- 高频 IP。
- 高频文章。

### 14.3 搜索统计

可选记录：

- 搜索关键词。
- 是否有结果。
- 点击了哪篇文章。

隐私边界：

- 不记录敏感个人信息。
- IP 尽量脱敏或限制可见角色。

## 15. 设计系统与 UI 规范

为了后续功能越加越多时不变乱，需要约束后台 UI。

### 15.1 后台视觉原则

- 后台是工作台，不做营销页式大 hero。
- 信息密度适中，优先可扫描。
- 卡片只用于独立信息块，不做卡片套卡片。
- 危险操作视觉上清晰但不夸张。
- 重要状态使用文字 + 颜色，不只依赖颜色。

### 15.2 组件规范

应沉淀这些通用组件：

- `AdminPageHeader`：标题、描述、主操作。
- `AdminToolbar`：筛选、搜索、批量操作。
- `AdminDataCard`：移动端列表卡片。
- `ConfirmDialog`：危险操作确认。
- `SheetDialog`：移动端编辑 sheet。
- `StatusBadge`：状态标签。
- `EmptyState`：空状态。
- `ErrorState`：错误状态。
- `LoadingSkeleton`：加载骨架。
- `FieldError`：字段级错误。
- `Toast`：全局轻提示。

### 15.3 表单规范

- label 永远可见。
- 错误提示靠近字段。
- 保存按钮有 loading。
- 保存失败保留用户输入。
- 长表单需要分组。
- 移动端保存按钮 sticky。

### 15.4 表格与卡片切换

- 桌面端可以使用表格。
- `<768px` 使用卡片。
- 不把宽表格硬塞进横向滚动容器，除非是审计日志这类天然宽数据。
- 移动端卡片必须展示主信息、状态、时间和主要操作。

## 16. 可访问性要求

后台功能越多，越需要可访问性底线。

必须满足：

- 所有图标按钮有 `aria-label`。
- modal / drawer 有 `role="dialog"` 和 `aria-modal="true"`。
- drawer / modal 打开后焦点不逃逸。
- ESC 可关闭非危险确认弹窗。
- 关闭后焦点回到触发按钮。
- 表单错误可被屏幕阅读器感知。
- 颜色对比度达到 WCAG AA。
- 键盘可以完成登录、搜索、发文、审评论。

移动端触控：

- 普通按钮最小 40px。
- 主要动作最小 44px。
- 危险操作和普通操作之间保留足够间距。

## 17. 性能策略

### 17.1 前台性能

- 首页避免一次加载所有文章。
- 图片懒加载。
- 文章封面使用稳定宽高比。
- Pagefind 资源加载失败要降级。
- 动效可被 `prefers-reduced-motion` 降级。
- 搜索、评论、登录这些 React islands 不应阻塞主体内容阅读。

### 17.2 后台性能

- 列表分页。
- 搜索防抖。
- 批量操作完成后局部刷新。
- 评论审核队列不一次加载全部历史评论。
- 媒体库懒加载缩略图。
- 仪表盘统计接口并发请求，但失败项独立显示。

### 17.3 构建性能

- `npm run build` 必须保持为常规验证命令。
- Pagefind 构建失败时要让构建失败，而不是静默缺搜索。
- 大图片不应进入前端源码目录导致构建变慢。

## 18. 错误处理与降级

每个核心模块都要有四种状态：

- 加载中。
- 空状态。
- 成功状态。
- 失败状态。

### 18.1 前台失败

- 文章不存在：给返回文章列表入口。
- 搜索失败：提示搜索暂不可用。
- 评论加载失败：给重试按钮。
- 评论提交失败：保留用户输入。

### 18.2 后台失败

- 权限不足：说明当前角色，并给返回入口。
- 保存失败：展示错误，并保留表单内容。
- 批量操作部分失败：显示成功数和失败数。
- 网络失败：允许重试，不清空列表状态。
- 认证失效：清理本地状态并跳转登录。

## 20. 发布与回滚流程

- `npm run build`。
- 移动端检查。

- 生产环境变量检查。
- 备份当前生产数据。
- 备份当前 `astro/dist`。

### 20.2 上线中

推荐流程：

- 本地构建。
- 打包 `astro/dist`。
- 上传服务器临时目录。
- 在服务器解压到新目录。
- 原子切换或替换静态资源目录。
- 重启 Caddy 或相关容器，避免 bind mount inode 问题。

### 20.3 上线后

检查：

- 首页。
- 登录页。
- 文章列表。
- 文章详情。
- 搜索。
- API health。
- 后台登录。
- 后台文章保存。
- 后台评论审核。

### 20.4 回滚

原则：

- UI 问题只回滚前端构建，不删除 PocketBase 数据卷。
- 迁移问题先停止后续写入，再按迁移方案处理。
- 不用删除 Docker volume 的方式解决前端问题。

## 22. 代码逻辑与安全自我验证体系

这里的“验证”不只指运行构建命令，也包括代码逻辑、权限边界、安全风险、数据一致性、移动端交互和部署风险的自我审查。每个功能完成后，都要先由实现者按本节做一轮自检，再进入人工验收或上线流程。

### 22.1 验证分层

每次改动至少按 6 层验证：

1. 代码逻辑验证：确认分支、状态、异常、边界输入都被处理。
2. 数据结构验证：确认字段、迁移、默认值、旧数据兼容。
3. 权限验证：确认前端入口、后台页面、PocketBase rules / hooks 三层一致。
4. 安全验证：确认没有密钥泄露、XSS、越权、绕过审核、危险操作缺确认。
5. 体验验证：确认桌面端、移动端、加载、失败、空状态都可用。
6. 部署验证：确认构建、迁移、备份、回滚、线上健康检查有闭环。

### 22.2 代码逻辑自我验证

每次实现功能后，逐项检查：

- 是否存在未处理的 `null` / `undefined` / 空数组。
- 是否存在只在理想数据下可用的逻辑。
- 是否处理了接口失败、超时、认证失效。
- 是否处理了重复点击、重复提交、并发保存。
- 是否处理了保存成功但局部刷新失败。
- 是否处理了批量操作的部分成功、部分失败。
- 是否有隐藏的状态不同步，例如本地状态成功但服务端失败。
- 是否有乐观更新，若有失败回滚逻辑是否明确。
- 是否有无限循环 `useEffect`、重复订阅、未清理监听器。
- 是否有 stale closure 导致保存旧数据。
- 是否有路由切换后继续 setState 的风险。
- 是否有分页、排序、筛选组合时的逻辑冲突。
- 是否有移动端和桌面端使用不同 DOM 时遗漏事件处理。

React / Astro 重点：

- React island 是否只在需要交互的地方使用。
- SSR 阶段是否访问了 `window`、`document`、`localStorage`。
- `client:load`、`client:visible`、`client:idle` 是否选择合理。
- Astro 页面获取数据失败时是否有降级 UI。
- `set:html` 使用前是否经过可信 sanitization。

### 22.3 数据一致性自我验证

数据结构或写操作变更时检查：

- 新字段是否有默认值。
- 旧记录缺字段时前端是否能正常显示。
- 迁移是否可重复验证。
- 新集合是否有最小权限规则。
- 删除对象前是否检查关联数据。
- 标签删除是否影响文章展示。
- 文章归档后是否从前台列表、搜索、相关文章中排除。
- 评论状态变化后 `public_comments` 是否一致。
- 批量操作是否能保证每条失败都有错误记录。
- 操作日志是否不会记录敏感明文。

PocketBase 迁移验证：

- 新迁移文件命名有时间戳。
- 迁移只做必要结构变更，不混入无关数据修补。
- 本地或临时 Docker 环境验证通过。
- 生产数据变更前有备份。

### 22.4 权限自我验证

所有后台功能都要验证“入口不可见”和“直接访问不可用”两件事。

角色矩阵：

| 功能 | reader | author | admin | super_admin |
| --- | --- | --- | --- | --- |
| 进入后台 | 否 | 是 | 是 | 是 |
| 管理自己的文章 | 否 | 是 | 是 | 是 |
| 管理所有文章 | 否 | 否 | 是 | 是 |
| 审核评论 | 否 | 否 | 是 | 是 |
| 管理标签 | 否 | 是 | 是 | 是 |
| 管理媒体 | 否 | 是 | 是 | 是 |
| 管理用户 | 否 | 否 | 否 | 是 |
| 修改站点设置 | 否 | 否 | 否 | 是 |
| 查看安全中心 | 否 | 否 | 否 | 是 |
| 查看操作日志 | 否 | 否 | 否 | 是 |
| 手动备份 | 否 | 否 | 否 | 是 |

自查问题：

- 菜单隐藏后，直接输入 URL 是否仍会被拦截？
- 前端按钮隐藏后，直接调用 API 是否仍会被 PocketBase 拒绝？
- `author` 是否只能改自己的文章？
- `admin` 是否不能提升用户为 `super_admin`？
- 最后一个 `super_admin` 是否不能被降级、禁用、删除？
- 普通注册是否永远只能得到 `reader`？
- 认证 token 过期后后台是否清理状态并跳转登录？

### 22.5 安全性自我验证

安全验证分为前端、后台、服务端和仓库四类。

前端安全：

- 前端代码中没有 API 密钥、SMTP 密钥、SSH 私钥、服务器密码。
- F12 干扰文案没有被描述成真实防护。
- 所有用户生成内容输出前经过清理或转义。
- 评论内容不使用未经清理的 HTML 注入。
- 外链自动添加 `rel="noopener noreferrer"`。
- 图片 URL、链接 URL 过滤危险协议，例如 `javascript:`。
- localStorage 中不存放额外敏感信息。

后台安全：

- 危险操作必须二次确认。
- 删除、角色变更、设置变更必须写操作日志。
- 操作日志不记录密码、token、完整私钥、完整环境变量。
- 安全中心只展示脱敏状态，不展示秘密原文。
- 批量删除必须显示影响数量和对象类型。

服务端安全：

- PocketBase rules 限制匿名读写边界。
- 评论创建规则不允许用户指定已审核状态。
- 用户角色写入由 hook 或 rules 保护。
- 管理后台入口由 Caddy / 反代限制。
- 登录接口有限流。
- 评论、举报、反应接口有限流。

仓库安全：

- `.env`、私钥、部署包、备份包被 `.gitignore` 忽略。
- 提交前检查是否出现 `PRIVATE KEY`、`PASSWORD`、`TOKEN`、`SECRET`、`sk-` 等敏感片段。
- 文档中不写真实服务器密码或完整密钥。
- 临时 SSH key 只放在忽略目录，部署后及时清理。

### 22.6 XSS 与内容清理验证

需要用恶意输入测试：

```html
<script>alert(1)</script>
<img src=x onerror=alert(1)>
<a href="javascript:alert(1)">click</a>
<svg onload=alert(1)></svg>
```

预期：

- 评论区不会执行脚本。
- 文章正文如果允许 HTML，只允许白名单标签和属性。
- 链接危险协议被移除或变成普通文本。
- 后台预览和前台展示清理逻辑一致。

### 22.7 业务流程自我验证

文章流程：

- 新建草稿。
- 自动保存。
- 刷新后恢复草稿。
- 发布文章。
- 前台可见。
- 撤回文章。
- 前台不可见。
- 归档文章。
- 归档文章不出现在列表、搜索、相关文章中。

评论流程：

- 未登录提交评论。
- 评论进入待审核。
- 前台不立即公开。
- 管理员通过。
- 前台展示。
- 管理员标记垃圾。
- 前台隐藏。
- 管理员删除。
- 操作日志可追踪。

权限流程：

- reader 访问 `/admin` 被拒绝。
- author 能写自己的文章。
- author 不能访问用户管理。
- admin 能审评论。
- admin 不能改站点设置。
- super_admin 能访问全部后台模块。

设置流程：

- 修改站点设置。
- 保存失败时保留输入。
- 保存成功后前台生效。
- 未保存离开时提示。

### 22.8 移动端逻辑验证

移动端不是只看是否“能显示”，还要验证交互逻辑：

- drawer 打开后背景不能滚动。
- drawer 关闭后焦点回到菜单按钮。
- 编辑 sheet 打开后底部保存按钮可见。
- 输入法弹出时保存按钮仍可访问。
- 批量选择后工具栏不遮挡内容。
- 横屏时 topbar 不遮住表单第一行。
- 文章编辑长正文时页面没有横向滚动。
- 评论卡片长邮箱、长 URL、长正文不撑破屏幕。

### 22.9 性能与资源验证

检查：

- 首页是否一次请求过多数据。
- 后台仪表盘多个统计失败时是否互不影响。
- 列表是否分页。
- 搜索是否防抖。
- 批量操作是否避免逐条阻塞 UI。
- React island 是否过度 `client:load`。
- 图片是否懒加载。
- 大图是否有稳定尺寸，避免布局跳动。

### 22.11 不通过标准

出现以下任一情况，不应上线：

- 构建失败。
- 迁移验证失败。
- 私钥、`.env`、部署包出现在 git 待提交列表。
- reader 能访问后台。
- author 能改别人的文章或用户角色。
- 未审核评论能直接公开。
- 删除文章、删除评论、修改角色没有确认。
- 后台保存失败会丢失用户输入。
- 手机端核心后台页面存在页面级横向滚动。
- 安全中心展示了完整密钥或环境变量。
